# EXP-022 Backlog (P0/P1/P2) з оцінками

База: `Docs/EXP022_SRS_IMPLEMENTATION_PLAN_EN.md`, `Docs/EXP022_SRS_IMPLEMENTATION_PLAN_UA.md`  
Одиниця оцінки: **інженерні дні** (чистий час розробки без зовнішніх затримок).

## P0 — Критично (обов’язково для робочого релізу)

| ID | Задача | Оцінка (дні) | Залежності | Критерій готовності |
|---|---|---:|---|---|
| P0-1 | Freeze baseline профілів/скриптів + фіксація KPI baseline | 1 | Немає | Є runbook baseline і файл метрик у репозиторії |
| P0-2 | Винести HID-логіку з `exp-022` в окремий HID-сервіс | 3 | P0-1 | HID сервіс запускається окремо, `exp-022` його споживає |
| P0-3 | Стабілізувати WS control контракт (`id/ok/error`, reconnect, timeout) | 2 | P0-2 | Немає “висячих” запитів, детермінований ACK/NACK |
| P0-4 | Нормалізувати taxonomy UART помилок (0x01..0x04, timeout, HMAC, busy) | 1.5 | P0-2 | UI/логи показують уніфіковані класи помилок |
| P0-5 | Реалізувати fallback через descriptor (`0x04`), якщо `0x05` layout відсутній/неповний | 4 | P0-2 | Mouse/keyboard працюють на non-boot/report-id пристроях |
| P0-6 | Довести коректність report build: report-id, signed/unsigned, variable length | 2.5 | P0-5 | Скриптові HID semantic тести >= 99% pass |
| P0-7 | Єдині diagnostics endpoints (`health/interfaces/layouts/last-reports`) | 2 | P0-2 | Endpoint’и віддають валідний стан активної сесії |
| P0-8 | Hardening AV baseline профілів (`balanced`, `low-latency`, `stability`) | 3 | P0-1 | 3 профілі відтворюються через конфіг/перемикання |
| P0-9 | 30-хв gate стабільності (без freeze >1s, без регулярних audio drop) | 2 | P0-8 | Gate-звіт пройдено 3 прогони підряд |

**Разом P0:** ~21 день

## P1 — Важливо (для сильної production готовності)

| ID | Задача | Оцінка (дні) | Залежності | Критерій готовності |
|---|---|---:|---|---|
| P1-1 | Runtime переключення AV профілів без рестарту | 3 | P0-8 | Профіль змінюється в сесії без обриву стріму |
| P1-2 | Єдина timeline AV+HID з correlation IDs | 2.5 | P0-7 | Є спільне хронологічне діагностичне представлення |
| P1-3 | Автоматичний збирач latency (clock helper + summary) | 2 | P0-8 | Avg/P95 latency експортуються після прогона |
| P1-4 | Тюнінг backpressure (черги/скидання по типах команд) | 2 | P0-3 | Mouse move контрольовано дропається, click/keys зберігаються |
| P1-5 | HID mapping packs (layout клавіатури, профілі per-target) | 2.5 | P0-6 | Mapping конфігурується і валідований на target |
| P1-6 | Recovery-тести (restart сервісу, network blip, короткий COM fault) | 2 | P0-7 | Поведінка відновлення задокументована і підтверджена |
| P1-7 | Експорт діагностичного пакету в 1 клік (JSON/log bundle) | 1.5 | P0-7 | Використовується для root-cause <5 хв |

**Разом P1:** ~15.5 дня

## P2 — Оптимізація (після RC)

| ID | Задача | Оцінка (дні) | Залежності | Критерій готовності |
|---|---|---:|---|---|
| P2-1 | Adaptive AV policy (рекомендації профілю з телеметрії) | 4 | P1-2, P1-3 | Рекомендації зменшують freeze rate у тестах |
| P2-2 | Розширений контроль якості (content-aware bitrate/GOP presets) | 3 | P0-8 | Кращий quality score в тому ж latency envelope |
| P2-3 | Розширений HID macro layer (безпечні шаблони shortcut/macro) | 2.5 | P1-5 | Макроси валідовані і мають rate-limit |
| P2-4 | Розширена matrix сумісності (додаткові OS/device варіанти) | 3 | P0-6 | Звіт сумісності з pass/fail по варіантах |
| P2-5 | Оптимізація продуктивності (CPU/GPU alerts + guardrails) | 2 | P1-2 | Сплески ресурсів детектяться і відображаються |

**Разом P2:** ~14.5 дня

## Пропозиція по мілстоунах

1. **M1 (P0 готово)**: ~4–5 календарних тижнів  
2. **M2 (P1 готово)**: +3 календарні тижні  
3. **M3 (P2 готово)**: +2–3 календарні тижні

(Для 1 основного інженера + part-time QA. Паралелізація зменшує календарний час.)
