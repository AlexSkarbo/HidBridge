# Підсумки EXP-022 по системі (AV + HID, ціль — мінімальна затримка)

## Мета

Реальна ціль EXP-022 — не тільки HID-керування.  
Це повний контур віддаленого доступу з:

- **максимально можливою якістю відео**
- **стабільним аудіо**
- **керуванням мишею/клавіатурою**
- **мінімальною end-to-end затримкою**

Цільова топологія:

`Браузер-клієнт <-> Сервер <-> Capture card (video/audio) + RP2040 HID bridge <-> Target PC`

## Що вже підтверджено

1. **WebRTC-плейбек працює** (WHEP/WHIP через SRS).
2. **Аудіо передається** за коректного Opus-профілю.
3. **Метод із годинником і синхронізацією працює** для наочного виміру лагу.
4. **HID-контур працює на транспортному рівні**:
   - Browser -> WS control -> UART -> RP2040
   - ACK стабільні.
5. **Семантика HID працює частково**:
   - Частина дій миші/клавіатури працює.
   - Частина варіантів report/layout ще потребує доробки.

## Ключові висновки (root causes)

1. **Затримка багатофакторна**, не тільки bitrate:
   - буферизація capture-пристрою
   - параметри FFmpeg (GOP, lookahead, B-frames, profile)
   - jitter-buffer браузера
   - черги/ретрансміти в SRS
   - навантаження CPU/GPU та scheduler

2. **Фрізи/підзависання** часто пов’язані з:
   - надто агресивним ultra-low-latency профілем при нестабільному source cadence
   - сплесками втрат/ретрансмітів (NACK storms)
   - занадто жорсткими параметрами енкодера відносно поточного навантаження

3. **Компроміс “якість vs затримка” реальний**:
   - екстремально низька затримка може погіршити якість та стабільність
   - збалансований low-latency профіль часто дає кращу реальну керованість

4. **HID ACK != реальна дія на target**:
   - UART ACK означає прийом кадру, але не гарантує коректний HID-ефект
   - критично важлива правильна побудова report (layout/descriptor/report-id)

## Оновлена ціль для production

Побудувати стабільну систему віддаленого керування з пріоритетами:

1. **Відгук керування** (передбачувана затримка, без втрат input)
2. **Безперервність AV** (без довгих фрізів, стабільне аудіо)
3. **Якість зображення** (максимум якості в межах бюджету затримки/стабільності)

## Рекомендоване розділення системи

1. **AV-сервіс**
   - capture + encode + publish профілі
   - runtime-перемикання профілів (quality/latency/stability)
   - AV-телеметрія (FPS, keyframe interval, packet loss, RTT, NACK/PLI)

2. **HID-сервіс**
   - WS/DataChannel вхід керування
   - UART-протокол адаптер
   - побудова report через layout + descriptor fallback
   - HID-телеметрія і семантична валідація

3. **Інтеграційний UI/стенд**
   - stream monitor + clock overlay
   - arm/disarm input, pointer lock
   - жива діагностика AV і HID

## Критерії успіху (рівень системи)

1. **Затримка**
   - зафіксувати цільовий envelope (приклад: median <= 200 ms у локальній LAN).

2. **Стабільність**
   - відсутність hard-freeze > 1 сек на 30-хв прогоні.
   - без систематичних обривів аудіо.

3. **Керування**
   - 99%+ успішних семантично коректних HID-дій у скриптових тестах.

4. **Якість**
   - базовий 1080p із прийнятним рівнем артефактів у межах цільової затримки.

## Наступний практичний план

1. Зафіксувати поточний baseline-профіль і журнал вимірів.
2. Додати єдину timeline-телеметрію для AV + HID.
3. Тюнити строго в порядку:
   - stability -> latency -> quality
4. Довести сумісність HID descriptor/layout.
5. Перейти до архітектури з двома сервісами (AV + HID), а `exp-022` лишити інтеграційним стендом.
